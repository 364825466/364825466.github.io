6. 解决Eureka启动失败问题以及日志输出优化

  • 首先解决eureka启动失败的问题

    在eureka的resources下新建application.properties

    spring.application.name=eureka
    server.port=8761
    eureka.client.fetch-registry=false
    eureka.client.register-with-eureka=false
    • 8761是eureka的默认端口
    • eureka.client.fetch-registry：获取注册信息，默认为true，因为我们本身就是注册中心了，所以改成false。
    • eureka.client.register-with-eureka：注册到一个eureka，默认为true，我们本身就是注册中心，所以改成false。

    然后在启动类中加入@EnableEurekaServer注解，就可以正常启动了

  • 然后美化我们的日志

    首先修改EurekaApplication启动类

    package com.course;

    import org.slf4j.Logger;
    import org.slf4j.LoggerFactory;
    import org.springframework.boot.SpringApplication;
    import org.springframework.boot.autoconfigure.SpringBootApplication;
    import org.springframework.cloud.netflix.eureka.server.EnableEurekaServer;
    import org.springframework.core.env.Environment;

    @SpringBootApplication
    @EnableEurekaServer
    public class EurekaApplication {

        // springboot默认继承slf4j框架，它定义了一些日志接口，规范日志输入，
    	// 真正的实现是logback或log4j,我们代码中写的都是slf4j的代码
    	private static final Logger LOG = LoggerFactory.getLogger(EurekaApplication.class);

    //	public static void main(String[] args) {
    //		SpringApplication.run(EurekaApplication.class, args);
    //	}

    	public static void main(String[] args) {
    		SpringApplication app = new SpringApplication(EurekaApplication.class);
    		Environment env = app.run(args).getEnvironment();
    		LOG.info("启动成功！！");
    		LOG.info("Eureka地址: \thttp://127.0.0.1:{}", env.getProperty("server.port"));
    	}

    }

    然后在resources下新建logback.xml，粘贴上去

    springboot默认用的日志框架是log back，我们要修改logback的输出格式就要新建logback.xml

    这个文件是一个比较固定的写法

    <?xml version="1.0" encoding="UTF-8"?>
    <configuration>
        <!-- 修改一下路径-->
        <property name="PATH" value="/logs/imooc/course/eureka"></property>
     <!-- 这个是项目所在盘符的根目录，就是比如项目在c盘，就是 c盘/logs.... -->
        
        <appender name="STDOUT" class="ch.qos.logback.core.ConsoleAppender">
            <encoder>
    <!--            <Pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} %highlight(%-5level) %blue(%-50logger{50}:%-4line) %msg%n</Pattern>-->
                <Pattern>%d{ss.SSS} %highlight(%-5level) %blue(%-30logger{30}:%-4line) %msg%n</Pattern>
            </encoder>
        </appender>

        <appender name="TRACE_FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
            <file>${PATH}/trace.log</file>
            <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
                <FileNamePattern>${PATH}/trace.%d{yyyy-MM-dd}.%i.log</FileNamePattern>
                <timeBasedFileNamingAndTriggeringPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedFNATP">
                    <maxFileSize>10MB</maxFileSize>
                </timeBasedFileNamingAndTriggeringPolicy>
            </rollingPolicy>
            <layout>
                <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} %-5level %-50logger{50}:%-4line %green(%-8X{UUID}) %msg%n</pattern>
            </layout>
        </appender>

        <appender name="ERROR_FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
            <file>${PATH}/error.log</file>
            <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
                <FileNamePattern>${PATH}/error.%d{yyyy-MM-dd}.%i.log</FileNamePattern>
                <timeBasedFileNamingAndTriggeringPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedFNATP">
                    <maxFileSize>10MB</maxFileSize>
                </timeBasedFileNamingAndTriggeringPolicy>
            </rollingPolicy>
            <layout>
                <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} %-5level %-50logger{50}:%-4line %green(%-8X{UUID}) %msg%n</pattern>
            </layout>
            <filter class="ch.qos.logback.classic.filter.LevelFilter">
                <level>ERROR</level>
                <onMatch>ACCEPT</onMatch>
                <onMismatch>DENY</onMismatch>
            </filter>
        </appender>

        <root level="ERROR">
            <appender-ref ref="ERROR_FILE" />
        </root>

        <root level="TRACE">
            <appender-ref ref="TRACE_FILE" />
        </root>

        <root level="INFO">
            <appender-ref ref="STDOUT" />
        </root>
    </configuration>

7. 搭建业务模块-system

新建system模块，步骤同上

接下来把system改造成springboot项目，在system的pom中加入依赖

<dependency>
          <groupId>org.springframework.cloud</groupId>
          <artifactId>spring-cloud-starter-netflix-eureka-client</artifactId>
      </dependency>

      <dependency>
          <groupId>org.springframework.boot</groupId>
          <artifactId>spring-boot-starter-web</artifactId>
      </dependency>

第一个依赖是eureka-client，我们新建的system要作为注册中心的一个客户端

第二个是springboot的基本的一个依赖

然后把eureka的启动类，application，log文件都拷过来

application.properties，名字改成system，端口号改一下

spring.application.name=system
server.port=9001

logback.xml，日志输出目录改一下

启动类名字改成SystemApplication

把Eureka的注解删除

现在就可以启动成功了，但是没什么结果，我们先建一个controller -> TestController

随便写个接口

import org.springframework.web.bind.annotation.RequestMapping;

@RestController
public class TestController {

    @RequestMapping("/test")
    public String test(){
        return "success";
    }
}

启动项目，访问test ， 看到success说明已经成功

8. 将system服务注册到注册中心

有三步

  1. 在system的pom中加入 eureka-client 依赖，之前已经加好了

  2. 增加配置，指向注册中心的地址

    eureka.client.service-url.defaultZone=http://localhost:8761/eureka/

  3. 在启动类中加入注解@EnableEurekaClient

现在再启动起来，打开Eureka的页面，可以看到端口号9001，已经注册了进来

9. 搭建网关gateway

网关的主要功能：

  • 限流：流量控制
  • 重试：请求失败时重试，慎用
  • 跨域：前后端不在一个域
  • 路由：转发请求
  • 鉴权：登录校验，签名校验等

搭建gateway模块

首先新建maven子模块，和之前一样

然后加两个依赖

一个网关gateway的依赖

一个注册到注册中心 eureka-client依赖

<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-gateway</artifactId>
</dependency>

<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-netflix-eureka-client</artifactId>
</dependency>

然后把system的启动类，application.properties，logback.xml都拷过来

  • 启动类改个名
  • application.properties
    • 名字改成gateway
    • 端口 9000，因为网关比较特殊，然后其他模块就9001这样往下排
  • logback 打印地址改成gateway

10 路由转发配置

路由转发：将外部请求转发到实际的业务模块进行处理

在gateway的application.properties中加入配置：

# 路由转发
spring.cloud.gateway.routes[0].id=system
spring.cloud.gateway.routes[0].uri=http://127.0.0.1:9001
spring.cloud.gateway.routes[0].predicates[0].name=Path
spring.cloud.gateway.routes[0].predicates[0].args[0]=/system/**

如果有多个 就把routes[0] 改成 routes[1] ,以此类推

  • 第一行是转发到哪一个业务模块（system）
  • 第二行是业务模块（system）所对应的地址
  • 后面，我们的路由时基于一个路径的，那么这个路径就是带/system/下面的任何路径都转发到这个路由上去

11 路由转发功能生效

现在system的test方法路径是

http://127.0.0.1:9001/test

http://127.0.0.1:9000/system/test是访问不到的

想让test前面加上/system能够访问到

在system的application.properties中加一行：

server.servlet.context-path=/system

现在访问http://127.0.0.1:9001/system/test就好使了

把端口改9000：

http://127.0.0.1:9000/system/test，也是好使的了

说明我们的路由转发功能生效了，原来9001的就不会暴露了，暴露的都是网关9000，这样外部只知道一个9000路由，别的不知道，上生产的时候我们可以把9001这个服务器拦截，让外网只能访问9000，之后也会在路由进行权限拦截，限流等功能。

总结

微服务基本的一个架构就是这样子，熟练之后可以先注册中心 -> gateway -> 业务模块
